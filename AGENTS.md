## OVERVIEW

FlashInfer AI Kernel Generation Contest starter kit (MLSys 2026). Template for implementing high-performance GPU kernels (Triton or CUDA) for LLM operations on NVIDIA Blackwell B200 GPUs, benchmarked via `flashinfer-bench`.

## STRUCTURE

```
flashinfer-kernel-gen/
├── config.toml                  # Track + language + entry point config (EDIT THIS FIRST)
├── solution/
│   ├── triton/kernel.py         # Triton kernel implementation (active when language=triton)
│   └── cuda/
│       ├── kernel.cu            # CUDA kernel implementation
│       └── binding.py           # TVM FFI Python bindings for CUDA
├── scripts/
│   ├── pack_solution.py         # Packs source → solution.json
│   ├── run_local.py             # Local GPU benchmark (needs FIB_DATASET_PATH)
│   └── run_modal.py             # Cloud benchmark on Modal B200 GPUs
├── mlsys26-contest/             # Git submodule: contest definitions + workloads (READ-ONLY)
│   ├── definitions/{track}/     # Kernel I/O specs + reference implementations (JSON)
│   └── workloads/{track}/       # Real inference workloads for benchmarking (JSONL)
├── solution.json                # Generated artifact (gitignored) — DO NOT EDIT
└── .env                         # FIB_DATASET_PATH environment variable
```

## CONVENTIONS

- **Destination Passing Style (DPS)**: Enabled by default. Kernel signature takes BOTH inputs AND pre-allocated outputs as parameters. If using value-returning style, set `destination_passing_style: false` in solution spec.
- **Entry point format**: `kernel.py::kernel` (file::function). Must match `config.toml` `entry_point`.
- **CUDA bindings**: Default `"tvm-ffi"`. Alternative `"torch"`. Set via `binding` field in spec.
- **No package structure**: `solution/` is NOT a Python package (no `__init__.py`). Files are packed into JSON.
- **Config-driven**: Language selection (triton/cuda) determines which `solution/` subdir is packed.

## ANTI-PATTERNS (THIS PROJECT)

- **DO NOT** use variadic input arguments in kernel signatures — fails builder validation
- **DO NOT** edit `solution.json` directly — it's generated by `pack_solution.py`
- **DO NOT** modify files in `mlsys26-contest/` — it's a read-only submodule
- **DO NOT** mismatch DPS: if kernel returns tensors instead of writing to pre-allocated params, you MUST set `destination_passing_style: false` or get `expected xx parameters, but got xx` error
- **DO NOT** forget `FIB_DATASET_PATH` env var — required for all benchmarking

## NOTES

- **Benchmark defaults**: warmup=3, iterations=100, trials=5
- **Python 3.12** required (specified in Modal image and conda setup)
- **GDN decode reference**: Uses float32 matmul for numerical stability, bfloat16 I/O, float32 state
- **GDN gate computation**: `g = exp(-exp(A_log) * softplus(a + dt_bias))`, `beta = sigmoid(b)`
- **Modal GPU**: B200:1 with 3600s timeout
- `solution.json` is in `.gitignore` but submission requires it committed or generated during eval
- The `mlsys26-contest/blob/` directory contains Git LFS objects (large binary workload data)
